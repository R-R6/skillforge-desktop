# 前后端契约规划器

> 从现有前端代码反推后端契约，生成页面数据需求、MySQL 5.7 建表草案、种子数据、OpenAPI 和联调说明。

支持平台: codex, cursor, claude, openclaw, hermes

## Prompt

# Frontend Contract Planner

Use this skill to read an existing frontend project and produce a conservative backend contract package. The default input is frontend source code only. Do not read UI design files unless the user explicitly asks for a UI-diff mode. The goal is not to invent backend behavior, but to document what the frontend proves, what it strongly implies, and what still needs confirmation.

## Operating stance

- Treat frontend code as evidence, not as complete business truth.
- Prefer static analysis over running the app. Running install, build, or dev commands is optional and only useful when static files are insufficient.
- Do not modify frontend source files. Write generated artifacts under the target project's `doc/backend-contract/` directory unless the user explicitly specifies another output path.
- If `doc/` or `doc/backend-contract/` does not exist, create it. Keep all generated contract files inside that directory so the project root stays clean.
- Make every table, field, endpoint, parameter, response field, enum, and permission traceable to a route, page, component, service, mock, type, store, constant, or existing API usage.
- Mark confidence explicitly. When evidence is weak, write the item as a draft and add the unresolved business rule to `05-assumptions-and-questions.md`.
- Keep the contract implementable by a backend engineer using MySQL 5.7 and REST OpenAPI 3.0.

## Inputs to inspect

Inspect these sources first, in this order:

1. Project metadata: `package.json`, build config, framework config, aliases, environment variable conventions.
2. Routing: route tables, file-based routes, menu config, permission routes, guards, layout nesting.
3. Pages: list/detail/create/edit pages, dialogs, drawers, import/export pages, batch operation surfaces.
4. API layer: request wrappers, service modules, generated API clients, interceptors, response normalization, error handling.
5. Mock and fixture data: mock servers, JSON fixtures, MSW handlers, local fake data, story data when it contains field values.
6. Types and schemas: TypeScript interfaces, Zod/Yup schemas, form validators, table column definitions.
7. State and constants: store modules, enums, dictionary options, permission codes, feature flags.
8. Existing docs only if they are colocated with the frontend and clearly describe the same pages.

Exclude by default:

- `node_modules`, `.git`, `.next`, `.nuxt`, `dist`, `build`, `coverage`, cache directories.
- Large lockfiles, minified bundles, source maps, generated assets, images, fonts, videos.
- Pure style files unless labels, field names, or configuration are only present there.
- Backend code or database migrations unless the user explicitly expands scope beyond frontend-only analysis.

## Workflow

### 1. Build the project profile

Identify the frontend framework, language, routing system, request abstraction, state manager, UI library, form library, table component patterns, mock locations, type locations, permission model, and path aliases.

Write the result to `00-overview.md` with:

- framework and version evidence,
- source directories inspected,
- excluded directories,
- request and response conventions,
- pagination conventions,
- authentication and permission evidence,
- confidence and unresolved gaps.

### 2. Create the page inventory

Generate `01-page-inventory.md` before designing entities. For each page record:

- page id and page name,
- route path,
- source file path,
- title or menu label,
- main business object,
- page mode: list, detail, create, edit, dashboard, workflow, settings, import/export, mixed,
- related components and dialogs,
- API/service calls,
- permission codes,
- confidence and evidence.

If a route is dynamic or generated, describe how it is discovered. If a page exists without a route, list it as an orphan candidate instead of dropping it.

### 3. Extract page requirements

Create one file per meaningful page under `doc/backend-contract/pages/{page-name}.md`. Use stable kebab-case English filenames derived from route or component names.

Each page file should contain:

- route, source files, page title, business object,
- query filters and search defaults,
- table/list columns,
- detail fields,
- create/edit form fields,
- validation rules visible in frontend,
- dialogs, drawers, tabs, steps, and nested tables,
- buttons and operations, including import, export, batch actions, status changes, delete, restore, submit, approve, reject,
- permission points and feature flags,
- API calls and data dependencies,
- field evidence table with source and confidence,
- page-specific assumptions and questions.

Use this extraction table shape:

| UI位置 | 字段名 | 页面文案 | 数据类型线索 | 是否必填 | 枚举/字典 | 来源 | 置信度 | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |

### 4. Merge entities conservatively

Build entity candidates by grouping page fields that clearly represent the same business object. Prefer names from route names, service names, type names, and domain terms already in the frontend.

When fields collide:

- Keep original evidence for each variant.
- Do not silently merge fields with different semantics.
- Add naming conflicts to `05-assumptions-and-questions.md`.
- Use aliases in documentation when the frontend uses multiple names for the same likely concept.

Classify fields as:

- persisted fields,
- request-only fields,
- response-only computed fields,
- UI-only fields,
- dictionary or enum candidates,
- relationship identifiers,
- audit and soft-delete fields.

## Output files

Generate this directory structure:

```text
doc/
  backend-contract/
    00-overview.md
    01-page-inventory.md
    02-database-schema.md
    03-api-contract.md
    04-integration-notes.md
    05-assumptions-and-questions.md
    schema.sql
    seed.sql
    openapi.yaml
    pages/
      {page-name}.md
```

If a file already exists, inspect it before editing and preserve useful user changes. Do not overwrite unrelated notes.

## Evidence and confidence rules

Use these confidence levels consistently:

- `High`: directly present in API service, mock response, type/interface, form schema, validation schema, table column config, or permission config.
- `Medium`: derived from route name, component name, label text, repeated field usage, or common frontend pattern with supporting evidence.
- `Low`: necessary contract placeholder with weak evidence. Add a question and avoid encoding business-specific behavior as fact.

Every generated artifact should contain evidence. For SQL, put detailed evidence in `02-database-schema.md`; keep SQL comments focused on Chinese business meaning and enum explanation.

## MySQL 5.7 schema rules

Generate `schema.sql` in MySQL 5.7 style:

- Use `utf8mb4` and `utf8mb4_unicode_ci`.
- Use `BIGINT UNSIGNED AUTO_INCREMENT` as the default primary key style.
- Use `DATETIME` for time fields. Avoid MySQL 8-only time expressions.
- Use `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci`.
- Avoid MySQL 8-only features: `CHECK` constraints, functional indexes, CTE-dependent designs, window-function-dependent designs, generated-column-dependent designs.
- Prefer explicit columns over `JSON`. Use MySQL 5.7 `JSON` only for genuinely semi-structured frontend payloads, and explain why in `02-database-schema.md`.
- Treat foreign keys as logical relationships by default. Do not create physical foreign keys unless the frontend or existing project convention strongly indicates that backend uses them.
- Add indexes for list filters, sort fields, unique business keys, logical relationship ids, and frequent status queries. Explain index purpose in `02-database-schema.md`.
- Include audit and soft-delete fields unless the frontend clearly uses another convention:
  - `created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'`
  - `updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'`
  - `created_by BIGINT UNSIGNED DEFAULT NULL COMMENT '创建人ID'`
  - `updated_by BIGINT UNSIGNED DEFAULT NULL COMMENT '更新人ID'`
  - `is_deleted TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否删除：0未删除，1已删除'`
  - `deleted_at DATETIME DEFAULT NULL COMMENT '删除时间'`

### Chinese SQL comments

All table and field comments in `schema.sql` must be Chinese. Use MySQL `COMMENT` clauses, not only `--` line comments.

Good field comments:

```sql
`status` TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '状态：0待确认，1启用，2停用',
`order_no` VARCHAR(64) NOT NULL COMMENT '订单编号',
`owner_user_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '负责人用户ID，逻辑关联用户表',
```

For enum fields, include known values in the Chinese comment. If enum values are uncertain, do not invent a final lifecycle. Use a neutral comment and add the uncertainty to `05-assumptions-and-questions.md`.

## Test data rules

After `schema.sql`, generate `seed.sql` with corresponding MySQL 5.7 compatible test data.

Seed data is synthetic contract data, not confirmed production truth. Make that explicit in the file header and in `04-integration-notes.md`.

Rules for `seed.sql`:

- Include `SET NAMES utf8mb4;`.
- Insert rows in dependency order so logical parent records exist before child records.
- Use explicit deterministic ids when rows need to reference each other.
- Prefer idempotent inserts with `ON DUPLICATE KEY UPDATE` when fixed primary keys are used.
- Generate enough rows to exercise page behavior: normally 3 to 8 rows per table, plus more when known enum states require coverage.
- Include at least one active row for normal list/detail flows.
- Include state variation only for enum values found in frontend constants, types, mocks, or UI options.
- Include one soft-deleted row when the page has delete, recycle-bin, restore, or `is_deleted` behavior.
- Do not create real credentials, tokens, API keys, phone numbers, ID card numbers, or personal data. Use obvious test placeholders.
- Ensure inserted columns match `schema.sql`; all `NOT NULL` columns without defaults must receive values.
- For uncertain fields, use conservative sample values and record the uncertainty in `05-assumptions-and-questions.md`.

Example seed style:

```sql
-- 本文件为前端契约推导的测试数据，仅用于联调和验收，不代表真实业务规则。
SET NAMES utf8mb4;

INSERT INTO `demo_order`
(`id`, `order_no`, `status`, `created_at`, `updated_at`, `is_deleted`)
VALUES
(1, 'TEST-ORDER-001', 0, '2026-01-01 10:00:00', '2026-01-01 10:00:00', 0)
ON DUPLICATE KEY UPDATE
`order_no` = VALUES(`order_no`),
`status` = VALUES(`status`),
`updated_at` = VALUES(`updated_at`),
`is_deleted` = VALUES(`is_deleted`);
```

## Database documentation

Write `02-database-schema.md` alongside `schema.sql`. For each table include:

- table name and Chinese table meaning,
- source pages and files,
- business purpose,
- field table with type, nullable, default, comment, source, confidence,
- enum and dictionary notes,
- logical relationships,
- indexes and why they exist,
- audit and soft-delete behavior,
- unresolved database questions.

Every table must map to at least one page, entity requirement, API usage, mock, type, or store requirement. If it does not, remove it or mark it as a question instead of generating it as fact.

## REST and OpenAPI rules

Generate REST-style contracts and `openapi.yaml` using OpenAPI 3.0.

For each endpoint, document:

- method and path,
- page operation that requires it,
- request path/query/body parameters,
- response schema,
- permission code or permission question,
- source table or logical entity,
- backend implementation notes,
- evidence and confidence.

Prefer shared endpoints when multiple pages use the same entity and operation. Avoid duplicating list/detail/create/update/delete contracts for every page if a single resource API covers them.

Use project conventions when proven by the frontend. If conventions are not proven, use conservative defaults:

- list: `GET /api/{resources}` with pagination and filters,
- detail: `GET /api/{resources}/{id}`,
- create: `POST /api/{resources}`,
- update: `PUT /api/{resources}/{id}`,
- delete or soft delete: `DELETE /api/{resources}/{id}`,
- batch operation: `POST /api/{resources}/batch-{action}`,
- export: `GET /api/{resources}/export`,
- import: `POST /api/{resources}/import`.

In `openapi.yaml`, add vendor extensions when useful:

```yaml
x-pages:
  - 用户列表
x-permissions:
  - system:user:list
x-source-tables:
  - sys_user
x-confidence: High
```

Keep `03-api-contract.md` and `openapi.yaml` consistent. The Markdown file is for humans; the YAML file is for tools.

## Integration notes

Write `04-integration-notes.md` for frontend and backend developers. Include:

- request base URL and auth header assumptions,
- response envelope convention,
- pagination convention,
- date/time format,
- file upload/download behavior,
- import/export behavior,
- error code and validation message expectations,
- permission-code mapping,
- frontend changes needed if backend contract differs from current mock/API assumptions,
- seed data usage notes.

## Assumptions and questions

Write `05-assumptions-and-questions.md` as a first-class artifact, not a dump at the end. Group open items by:

- business rules,
- enum and status lifecycle,
- permission boundaries,
- validation rules,
- database relationships,
- unique constraints,
- import/export format,
- audit and soft-delete policy,
- fields or pages with low confidence,
- generated test data assumptions.

Each question should include why it matters and which pages or endpoints are blocked by it.

## Consistency check

Before finishing, check:

- Every route page is either documented or explicitly excluded with a reason.
- Every page operation has an API contract or a pending question.
- Every API in `03-api-contract.md` appears in `openapi.yaml` with the same method and path.
- Every request and response field has evidence or an explicit assumption.
- Every table in `schema.sql` appears in `02-database-schema.md`.
- Every table maps to at least one page, mock, type, store, or API requirement.
- `schema.sql` contains no MySQL 8-only syntax.
- Every generated table and field comment in `schema.sql` is Chinese.
- `seed.sql` inserts valid columns that exist in `schema.sql`.
- `seed.sql` covers normal list/detail data and known enum states without inventing unproven lifecycle rules.
- All low-confidence items are present in `05-assumptions-and-questions.md`.

## Final response

When delivering the work, summarize:

- where `doc/backend-contract/` was written,
- pages and entities discovered,
- number of tables, endpoints, and seed-data rows generated,
- highest-risk assumptions,
- whether validation passed,
- any commands run.

Do not claim backend implementation is complete. This skill produces a contract package for backend implementation and frontend-backend alignment.
